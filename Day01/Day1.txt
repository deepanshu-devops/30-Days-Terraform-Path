🚀 Day 01 — What is Terraform & Why Infrastructure as Code?
I am starting my 30-day Terraform journey! Today is all about the fundamentals. Here’s the breakdown:
🔍 THE 5W + 1H FRAMEWORK
👥 WHO: DevOps Engineers, Platform Engineers, SREs, and Cloud Architects.
📦 WHAT: An open-source IaC tool by HashiCorp using HCL (HashiCorp Configuration Language). It manages infra via machine-readable config files instead of manual clicks.
⏰ WHEN: Use it for provisioning cloud resources, creating identical environments (Dev/Prod), and ensuring disaster recovery.
📍 WHERE: Runs locally or in CI/CD (GitHub Actions/Jenkins). It manages AWS, Azure, GCP, Kubernetes, and 3,000+ other providers.
💡 WHY:
• Manual: Click-driven, error-prone, no audit trail.
• Terraform: Code-driven, repeatable, full Git history.
🛠️ HOW: Write (.tf files) ➡️ Init ➡️ Plan ➡️ Apply ➡️ Destroy(If needed).
------------------------------
🧠 AUDIENCE-LEVEL EXPLANATIONS
🟢 Beginner: It's a recipe. Instead of clicking "Create VPC," you write the code, and Terraform "cooks" it.
🔵 Intermediate: It’s declarative. You describe "what" you want, and Terraform figures out the "how" using a dependency graph.
🟠 Advanced: It manages State Drift. It diffs desired state (config) vs. known state (.tfstate) vs. real state (APIs).
🔴 Expert: It uses a gRPC interface for providers and handles large-scale concerns like state partitioning and provider version pinning.
------------------------------
📝 KEY CONCEPTS SUMMARY
🔹 HCL: The syntax for .tf files.
🔹 Provider: The plugin for a cloud API who fill the gap between tf binary and cloud API's.
🔹 Resource: The infra piece (VPC, EC2, ETC....).
🔹 State: The record of managed infra.
🔹 Plan/Apply: Preview and execute changes.
------------------------------
⚠️ COMMON MISCONCEPTIONS

   1. "Terraform is only for AWS" (False: It supports 3,000+ providers).
   2. "It manages everything automatically" (False: Day-2 ops still need orchestration).
   3. "Once applied, infra is safe" (False: Manual changes can still cause drift).

------------------------------
✅ PRODUCTION BEST PRACTICES
• Never store terraform.tfstate locally for team projects.
• Always run terraform plan before apply.
• Store all .tf files in Git (The Source of Truth).
