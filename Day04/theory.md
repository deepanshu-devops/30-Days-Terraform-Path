# Day 04 — The Terraform Lifecycle: init → plan → apply → destroy

## Real-Life Example 🏗️

**Scenario:** Your QA team asks for a fresh test environment every Monday and wants it torn down every Friday to save costs.

**Without Terraform:**
- Monday: DevOps engineer manually provisions 6 AWS resources → 45 minutes
- Friday: Manually finds and deletes them → hopes nothing is missed → leftover charges appear in billing

**With Terraform:**
```bash
# Monday morning — 3 minutes
terraform init && terraform apply -auto-approve

# Friday evening — 90 seconds
terraform destroy -auto-approve
```

Automated, repeatable, zero forgotten resources, zero surprise billing.

---

## The Four Core Commands

### `terraform init` — "Get Ready"

Downloads providers and modules. Must run first and every time you add a new provider or module.

```bash
terraform init

# What happens internally:
# 1. Reads required_providers in provider.tf
# 2. Downloads providers to .terraform/providers/
# 3. Creates/updates .terraform.lock.hcl with exact checksums
# 4. Configures the state backend

# Useful flags:
terraform init -upgrade           # upgrade providers within version constraints
terraform init -reconfigure       # force re-init of the backend
terraform init -migrate-state     # move state from old backend to new backend
terraform init -backend=false     # skip backend setup (useful in CI validation)
```

**Analogy:** `npm install` — sets up all dependencies before you run anything.

> ✅ **Commit `.terraform.lock.hcl` to Git** — it pins exact provider versions for the whole team.
> ❌ **Never commit `.terraform/`** — it's large, machine-specific, regenerated by init.

---

### `terraform plan` — "Show Me What Will Change"

Compares desired state (your `.tf` files) against known state (`.tfstate`) and real infra (live AWS API calls). Shows exactly what will be created, changed, or destroyed — without touching anything.

```bash
terraform plan

# Save the plan for a deterministic apply:
terraform plan -out=tfplan.binary

# Read a saved plan:
terraform show tfplan.binary

# Preview what destroy would do:
terraform plan -destroy

# Plan only one resource (use sparingly):
terraform plan -target=aws_vpc.main

# Override a variable:
terraform plan -var="environment=staging"
```

**Reading plan output:**
```
Terraform will perform the following actions:

  # aws_vpc.main will be created
  + resource "aws_vpc" "main" {
      + cidr_block           = "10.0.0.0/16"   # green = new value
      + id                   = (known after apply)
    }

  # aws_subnet.public will be updated in-place
  ~ resource "aws_subnet" "public" {
      ~ tags = {
          + "Env" = "staging"                    # yellow ~ = changed
        }
    }

  # aws_instance.old will be destroyed
  - resource "aws_instance" "old" {              # red - = removed
    }

  # aws_instance.new must be replaced
  -/+ resource "aws_instance" "new" {            # red -/+ = destroy + recreate
      ~ ami = "ami-old" -> "ami-new" # forces replacement
    }

Plan: 1 to add, 1 to change, 1 to destroy, 1 to replace.
```

> **Golden rule:** Never skip `plan`. A senior engineer once ran `apply` on prod without checking plan. An RDS instance was destroyed. They spent the weekend restoring from backup.

---

### `terraform apply` — "Make It Happen"

Executes the plan and makes real changes to AWS.

```bash
terraform apply                    # runs plan, asks "Do you want to perform these actions?"
terraform apply -auto-approve      # skips the prompt (CI/CD pipelines only)
terraform apply tfplan.binary      # applies the exact saved plan (deterministic)

# Useful flags:
terraform apply -parallelism=20    # 20 concurrent operations (default: 10)
terraform apply -target=aws_vpc.main  # apply only one resource
terraform apply -replace=aws_instance.web  # force recreate one resource
```

> **In production CI/CD:** Never use `-auto-approve` without a human approval gate. The pipeline should post the plan as a PR comment → engineer reviews → merge → pipeline applies.

---

### `terraform destroy` — "Tear It All Down"

Destroys all resources managed by this configuration.

```bash
terraform destroy                  # shows destroy plan, asks for confirmation
terraform destroy -auto-approve    # skips confirmation (careful!)

# Safer approach — review the destroy plan first:
terraform plan -destroy -out=destroy.plan
terraform show destroy.plan        # review what will be deleted
terraform apply destroy.plan       # destroy the exact reviewed plan

# Destroy only one resource:
terraform destroy -target=aws_instance.test
```

**When to use destroy:**
- ✅ Tear down dev/test environments at sprint end
- ✅ Clean up PR preview environments after merge
- ✅ Remove a temporary debugging environment
- ❌ Never on production without an explicit plan review and approval

## The Complete Daily Workflow

```
1.  Edit .tf files
2.  terraform fmt          ← always run this first (auto-formats)
3.  terraform validate     ← catch syntax errors (no AWS calls)
4.  terraform plan -out=tfplan
5.  Review plan output carefully
6.  terraform apply tfplan
7.  terraform output       ← verify the result
```

## Common Mistakes

| Mistake | What Happens | Prevention |
|---------|-------------|-----------|
| Skip `terraform init` after adding a provider | "Could not load plugin" error | Always run init after changing providers |
| Apply without reviewing plan | Unexpected resource deletion | Always review plan output |
| `apply -auto-approve` on prod | No chance to catch mistakes | Require human approval in CI/CD |
| `destroy` on prod manually | Catastrophic loss | Add `prevent_destroy = true` on critical resources |
| Forget to commit `.terraform.lock.hcl` | Team uses different provider versions | Commit the lock file |
